## Elasticsearch Write Path Internals: Refresh vs Flush vs Translog

> One of the most asked Elasticsearch internals topics — and one where most people mix up the terms.
> Understanding this end-to-end flow is critical for tuning performance and understanding data durability.

---

## 🔁 The Complete Write Path — End to End

```
Client Write Request
        │
        ▼
┌───────────────────┐
│   In-Memory       │  ← Document lands here first (In-memory buffer)
│   Buffer          │     NOT searchable yet
└───────┬───────────┘
        │
        │  simultaneously written to ↓
        │
┌───────────────────┐
│   Translog        │  ← Write-Ahead Log (WAL) on DISK
│   (on disk)       │     Ensures durability before segment is created
└───────┬───────────┘
        │
        │  REFRESH happens (every 1 sec by default)
        ▼
┌───────────────────┐
│  In-memory        │  ← Buffer converted to in-memory Lucene Segment
│  Segment          │     NOW searchable! But NOT yet on disk
│  (searchable)     │
└───────┬───────────┘
        │
        │  More writes → More small in-memory segments build up
        │
        │  FLUSH happens (translog size threshold / time based)
        ▼
┌───────────────────┐
│  Segment on DISK  │  ← Segment written to disk via fsync
│  (persisted)      │     Translog is cleared after this
└───────────────────┘
        │
        │  Over time → too many small segments
        │
        │  MERGE happens (automatic by Lucene)
        ▼
┌───────────────────┐
│  Merged Segment   │  ← Small segments combined into larger ones
│  on DISK          │     Old segments deleted
└───────────────────┘
```

---

## 📖 Three Concepts Explained Clearly

### 1. 🔄 Refresh

- **What:** Converts in-memory buffer → new in-memory Lucene segment
- **Result:** Document becomes **searchable**
- **Default interval:** Every **1 second** (`index.refresh_interval`)
- **Does it hit disk?** ❌ NO — segment is still in memory
- **Cost:** Relatively cheap but not free — avoid too frequent refreshes on write-heavy indices

```bash
# Manual refresh
POST my-index/_refresh

# Disable refresh during bulk indexing (re-enable after)
PUT my-index/_settings
{
  "index": {
    "refresh_interval": "-1"
  }
}

# Re-enable after bulk load
PUT my-index/_settings
{
  "index": {
    "refresh_interval": "1s"
  }
}
```

---

### 2. 📝 Translog (Transaction Log)

- **What:** A write-ahead log — every document write is immediately appended here
- **Lives on:** DISK (not memory)
- **Purpose:** **Durability** — if node crashes before flush, translog replays the writes on recovery
- **Cleared when:** After a successful **flush** (segments are safely on disk)
- **Think of it like:** A bank's transaction ledger — every operation recorded before final settlement

```bash
# Check translog stats
GET my-index/_stats/translog

# Translog durability setting
PUT my-index/_settings
{
  "index": {
    "translog.durability": "request",   # fsync after every request (safe, slower)
    "translog.durability": "async"      # fsync every sync_interval (faster, slight risk)
  }
}
```

---

### 3. 🚿 Flush

- **What:** Writes in-memory segments to disk via **fsync** + clears the translog
- **Result:** Data is **durably persisted** on disk
- **Triggered by:**
  - Translog reaching size threshold (default 512MB)
  - Time interval (`index.translog.flush_threshold_period` default 30 min)
  - Manual flush
- **Cost:** Expensive I/O operation — don't do it too frequently

```bash
# Manual flush
POST my-index/_flush

# Check flush stats
GET my-index/_stats/flush
```

---

## ⚡ Refresh vs Flush — The Key Difference

| | Refresh | Flush |
|---|---|---|
| **Purpose** | Make data searchable | Persist data to disk |
| **Writes to disk?** | ❌ No | ✅ Yes (fsync) |
| **Clears translog?** | ❌ No | ✅ Yes |
| **Default frequency** | Every 1 second | When translog hits 512MB |
| **Performance cost** | Low-Medium | High (I/O heavy) |
| **After node crash** | Data in memory = lost | Data on disk = safe |

## 🧠 Memory Thresholds — What Triggers What

```
Document Indexed
      │
      ├──→ Written to Translog (disk) immediately
      │
      └──→ Sits in In-Memory Buffer
                  │
                  │  refresh_interval = 1s (default)
                  ▼
         Becomes searchable segment in memory
                  │
                  │  translog.flush_threshold_size = 512mb (default)
                  │  OR translog.flush_threshold_period = 30m
                  ▼
         Segment fsynced to disk → Translog cleared
                  │
                  │  Lucene auto-merge policy kicks in
                  ▼
         Small segments merged into larger segments
         (old segments deleted)
```

## 🛠️ Real-World Tuning Tips

**Scenario 1: Bulk Indexing (high write throughput)**
```bash
# Disable refresh + increase translog threshold
PUT my-index/_settings
{
  "index": {
    "refresh_interval": "-1",
    "translog.flush_threshold_size": "1gb"
  }
}
# Re-enable after bulk load done
```

**Scenario 2: Near Real-Time search needed**
```bash
# Keep refresh at 1s or reduce it
PUT my-index/_settings
{
  "index": {
    "refresh_interval": "1s"
  }
}
```

**Scenario 3: After restore / migration — force flush**
```bash
POST my-index/_flush
POST my-index/_forcemerge?max_num_segments=1
```

---

## 💡 Key Takeaway

> **Refresh = Searchability** (memory only)

> **Translog = Durability** (disk, crash safety)

> **Flush = Persistence** (memory → disk, translog cleared)

>
> A document can be **searchable but not yet durable** (after refresh, before flush).
> A document is **durable but perhaps not yet searchable** only in edge cases.
> The translog bridges this gap — ensuring no data loss between refresh and flush.
