## 🧠 The Big Picture — How Everything Connects

```
COMPONENT TEMPLATES          INDEX TEMPLATE           DATA STREAM
(reusable building blocks)   (blueprint)              (logical view)
                                                            │
┌──────────────────┐         ┌──────────────────┐         │
│ mappings-        │────────▶│                  │         │
│ component        │         │  logs-template   │────────▶│ logs-myapp-prod
│                  │         │                  │         │      │
│ settings-        │────────▶│  index_patterns: │         │      ▼
│ component        │         │  ["logs-*-*"]    │    ┌────┴──────────────────┐
│                  │         │                  │    │    Backing Indices     │
│ ilm-component    │────────▶│  data_stream: {} │    │                       │
└──────────────────┘         └──────────────────┘    │ .ds-logs-myapp-prod   │
                                      │               │   -2026.04.01-000001  │
                                      │               │ .ds-logs-myapp-prod   │
                              ILM POLICY               │   -2026.04.27-000002  │◀─ write index
                              (lifecycle rules)        └───────────────────────┘
                                      │                        │
                              ┌───────▼────────┐               │
                              │   hot phase    │  rollover ────┘
                              │   warm phase   │
                              │   cold phase   │
                              │   delete phase │
                              └────────────────┘
```

### The Sequence — Always Follow This Order

```
Step 1 → Create ILM Policy
Step 2 → Create Component Templates (mappings, settings, ilm)
Step 3 → Create Index Template (compose component templates)
Step 4 → Create Data Stream (first write auto-creates it)
Step 5 → ILM manages rollover + tier migration automatically
```

## 📐 Part 1 — ILM Policy (Create This First!)

### What is ILM?

Index Lifecycle Management automatically manages indices through phases:
- **Hot** → Active indexing + search, rollover when size/age limit hit
- **Warm** → No more indexing, optimized for search (shrink + forcemerge)
- **Cold** → Infrequent search, reduced resources
- **Frozen** → Cheapest storage, on-demand access via searchable snapshots
- **Delete** → Index removed

### Production ILM Policy — Logs Use Case

```bash
PUT _ilm/policy/logs-production-policy
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_primary_shard_size": "50gb",
            "max_age": "1d",
            "max_docs": 200000000
          },
          "set_priority": {
            "priority": 100
          }
        }
      },
      "warm": {
        "min_age": "3d",
        "actions": {
          "migrate": {},
          "shrink": {
            "number_of_shards": 1
          },
          "forcemerge": {
            "max_num_segments": 1
          },
          "set_priority": {
            "priority": 50
          }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "migrate": {},
          "set_priority": {
            "priority": 0
          }
        }
      },
      "frozen": {
        "min_age": "90d",
        "actions": {
          "searchable_snapshot": {
            "snapshot_repository": "my-s3-repo",
            "force_merge_index": true
          }
        }
      },
      "delete": {
        "min_age": "365d",
        "actions": {

  }
}
```

## 🧩 Part 2 — Component Templates

### What are Component Templates?

Reusable building blocks that are **composed** into index templates.
Instead of repeating mappings/settings across 10 templates — define once, reuse everywhere.

```
Think of it like: functions in programming
Component Template = function definition
Index Template     = function call with composition
```

## 📄 Part 3 — Index Templates

### What is an Index Template?

A blueprint that automatically applies settings, mappings, and aliases
to any new index/data stream whose name matches the pattern.

```
Index Template = composer of Component Templates + data_stream config + priority
```

### Production Index Template — Logs

```bash
PUT _index_template/logs-production-template
{
  "index_patterns": ["logs-*-*"],
  "data_stream": {},
  "composed_of": [
    "logs-mappings",
    "logs-settings",
    "logs-ilm-settings"
  ],
  "priority": 500,
  "template": {
    "settings": {
      "number_of_shards": 1,
      "number_of_replicas": 1
    },
    "mappings": {
      "_source": {
        "enabled": true
      }
    },
    "lifecycle": {
      "data_retention": "365d"
    }
  },
  "_meta": {
    "description": "Production template for all log data streams",
    "owner": "platform-team",
    "version": "1.0.0"
  }
}
```

### Production Index Template — Nginx (extends logs template)

```bash
PUT _index_template/logs-nginx-template
{
  "index_patterns": ["logs-nginx-*"],
  "data_stream": {},
  "composed_of": [
    "logs-mappings",
    "logs-http-mappings",
    "logs-settings",
    "logs-ilm-settings"
  ],
  "priority": 600,
  "_meta": {
    "description": "Nginx access + error logs — inherits HTTP mappings",
    "version": "1.0.0"
  }
}
```

### Priority — How Conflicts Are Resolved

```
Higher priority number wins when patterns overlap.

logs-nginx-*  → priority 600  (more specific — wins for nginx)
logs-*-*      → priority 500  (catch-all)
logs-*        → priority 100  (lowest — built-in/fallback)

So for index "logs-nginx-prod":
  → Both templates match
  → priority 600 wins → logs-nginx-template is applied
```

## 🌊 Part 4 — Data Streams

### What is a Data Stream?

A Data Stream is a logical abstraction over multiple time-series backing indices.

```
Data Stream: logs-myapp-prod
    │
    ├── .ds-logs-myapp-prod-2026.04.01-000001  (old, read-only after rollover)
    ├── .ds-logs-myapp-prod-2026.04.10-000002  (old, read-only after rollover)
    └── .ds-logs-myapp-prod-2026.04.27-000003  ← write index (current)
```

### Rules for Data Streams

```
1. ALL documents MUST have @timestamp field
2. All writes go to the current write index (latest backing index)
3. Reads (search) query ALL backing indices simultaneously
4. Rollover creates a new backing index → old one becomes read-only
5. Data stream name must match index template pattern
6. Naming convention: {type}-{dataset}-{namespace}
   e.g., logs-nginx-production, metrics-system-prod, traces-apm-staging
```

### Create Data Stream

```bash
# Method 1: Auto-create on first document write
# (if matching index template exists with data_stream: {})
POST logs-myapp-prod/_doc
{
  "@timestamp": "2026-04-27T10:30:00.000Z",
  "message": "Application started",
  "service.name": "myapp",
  "log.level": "INFO"
}
# → Data stream "logs-myapp-prod" created automatically!

# Method 2: Explicit creation
PUT _data_stream/logs-myapp-prod

# Verify it was created
GET _data_stream/logs-myapp-prod
```

### Manual Rollover (force new backing index)

```bash
# Manual rollover (creates new write index)
POST logs-myapp-prod/_rollover

# Rollover with conditions (only rolls if conditions met)
POST logs-myapp-prod/_rollover
{
  "conditions": {
    "max_age": "7d",
    "max_docs": 1000000,
    "max_primary_shard_size": "50gb"
  }
}

# Check generation after rollover
GET _data_stream/logs-myapp-prod
# generation number increments with each rollover
```

```bash

    }
  }

    "number_of_replicas": 2
  }

```


## 🚨 Part 7 — Common Mistakes & Fixes

| Mistake | Impact | Fix |
|---|---|---|
| Creating data stream before index template | Data stream uses default mappings | Create template first, then data stream |
| Two templates same priority, overlapping patterns | Unpredictable template applied | Always use unique priorities |
| ILM policy not attached to template | Indices grow forever, never roll over | Always include `index.lifecycle.name` in component template settings |
| `dynamic: strict` but not all fields mapped | Documents with unknown fields rejected | Use `dynamic: false` for unknown fields or map all fields |
| Forgetting `@timestamp` in documents | Data stream rejects document | Always include `@timestamp` |
| Changing shard count on existing index | Not possible without reindex | Change in template + rollover for new backing indices |
| Using same template priority for different patterns | Conflict — last one wins alphabetically | Assign priority 100/200/300/400 based on specificity |
| Deleting component template used by index template | Index template broken silently | Check usage before deleting: `GET _index_template` |
| Manual rollover breaking ILM | ILM confused about lifecycle step | Use `POST logs-*/_rollover` and let ILM manage |

## 💡 Key Principles — Summary

```
1. Order matters: ILM → Component Templates → Index Template → Data Stream

2. Component templates = DRY principle
   Define mappings/settings ONCE, reuse across 10+ templates

3. Priority resolves conflicts
   Higher number = higher priority. Specific templates always > catch-all

4. Data stream naming: {type}-{dataset}-{namespace}
   logs-nginx-production, metrics-system-staging, traces-apm-dev

5. Simulate before applying
   POST _index_template/_simulate_index/my-index-name
   Always test before writing real data

6. @timestamp is mandatory
   Data streams reject documents without it

7. Rollover is non-destructive
   Old backing indices go read-only, data stays safe

8. ILM policy changes are live
   Update policy → existing indices pick up new rules on next check (10 min)

9. Never manually name backing indices
   Let Elasticsearch manage .ds-* naming automatically

10. Component template changes apply on next rollover
    Existing backing indices keep old settings — new backing index gets new settings
```
