## 🧠 Core Concepts — Understand This First

```
SNAPSHOT
  └── Point-in-time backup of one or more indices/data streams
  └── Incremental — only changed segments are stored after first snapshot
  └── Stored in a REPOSITORY (S3, GCS, Azure, HDFS, shared filesystem)

REPOSITORY
  └── Storage backend where snapshots live
  └── One repository can hold MANY snapshots
  └── Multiple clusters can READ the same repository (read-only)
  └── Only ONE cluster should WRITE to a repository at a time

SLM (Snapshot Lifecycle Management)
  └── Automates snapshot creation, retention, deletion
  └── Like cron job for snapshots
  └── Manages retention — auto-deletes old snapshots

SEARCHABLE SNAPSHOTS
  └── Mount snapshot as a live index — WITHOUT restoring to local disk
  └── Data stays in object storage (S3/GCS)
  └── Queries fetch data on demand
  └── Used for COLD and FROZEN tiers — massive cost saving
```

### How Snapshots Work Internally

```
First Snapshot (full):
  Index segments → ALL segments copied to repository
  Snapshot size ≈ Index size

Second Snapshot (incremental):
  Only NEW/CHANGED segments copied
  Unchanged segments referenced from previous snapshot
  Snapshot size ≈ Only delta

Delete old snapshot:
  Segments still referenced by newer snapshots → KEPT
  Segments not referenced by any snapshot → DELETED
  (Smart reference counting — never orphans data)
```

## 🗄️ Part 1 — Repository Types & Setup

### Type 1: S3 Repository (Most Common in Production)

```bash
# Step 1: Verify plugin is installed (included by default in 8.x+)
GET _cat/plugins

# Step 2: Store credentials in keystore (never in API!)
# Run on EVERY node:
bin/elasticsearch-keystore add s3.client.default.access_key
bin/elasticsearch-keystore add s3.client.default.secret_key

# OR use IAM Role (recommended for AWS) — no keys needed!
# Attach IAM role to EC2/ECS with S3 permissions

# Step 3: Register S3 repository
PUT _snapshot/s3-production-repo
{
  "type": "s3",
  "settings": {
    "bucket": "my-es-snapshots-prod",
    "region": "ap-south-1",
    "base_path": "elasticsearch/production",
    "client": "default",
    "server_side_encryption": true,
    "storage_class": "standard",
    "compress": true,
    "chunk_size": "1gb",
    "max_restore_bytes_per_sec": "500mb",
    "max_snapshot_bytes_per_sec": "500mb",
    "readonly": false
  }
}

# Step 4: Verify repository
POST _snapshot/s3-production-repo/_verify

# Step 5: Analyze repository (health check)
POST _snapshot/s3-production-repo/_analyze?blob_count=10&max_blob_size=1mb&timeout=120s
```

### IAM Policy for S3 Repository (Minimum Required)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetBucketLocation",
        "s3:ListBucketMultipartUploads",
        "s3:ListBucketVersions"
      ],
      "Resource": "arn:aws:s3:::my-es-snapshots-prod"
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": "arn:aws:s3:::my-es-snapshots-prod/*"
    }
  ]
}
```

## 📸 Part 2 — Taking Snapshots

### Basic Snapshot

```bash
# Snapshot ALL indices + cluster state
PUT _snapshot/s3-production-repo/snapshot-2026-04-27
{
  "indices": "*",
  "include_global_state": true,
  "metadata": {
    "taken_by": "vinay",
    "reason": "pre-upgrade backup"
  }
}

# Snapshot specific indices only
PUT _snapshot/s3-production-repo/snapshot-logs-2026-04-27
{
  "indices": ["logs-*", "metrics-*"],
  "include_global_state": false,
  "ignore_unavailable": true,
  "metadata": {
    "taken_by": "slm-policy",
    "env": "production"
  }
}

# Snapshot a data stream
PUT _snapshot/s3-production-repo/snapshot-ds-2026-04-27
{
  "indices": ["logs-nginx-production", "logs-api-production"],
  "feature_states": [],
  "include_global_state": false
}
```

### Monitor Snapshot Progress

```bash
# Check snapshot status (while running)
GET _snapshot/s3-production-repo/snapshot-2026-04-27/_status

# Response shows:
# state: IN_PROGRESS | SUCCESS | FAILED | PARTIAL
# stats.total.size_in_bytes
# stats.processed.size_in_bytes
# shards_stats.done / total

# Get snapshot info (after complete)
GET _snapshot/s3-production-repo/snapshot-2026-04-27

# List ALL snapshots in repository
GET _snapshot/s3-production-repo/_all

# List latest 5 snapshots
GET _snapshot/s3-production-repo/_all?size=5&sort=start_time&order=desc

# Currently running snapshots
GET _snapshot/_status
```

### Delete Snapshots

```bash
# Delete single snapshot
DELETE _snapshot/s3-production-repo/snapshot-2026-04-01

# Delete multiple snapshots
DELETE _snapshot/s3-production-repo/snapshot-2026-04-01,snapshot-2026-04-02

# Delete by pattern (use SLM retention instead in production)
DELETE _snapshot/s3-production-repo/snapshot-2026-03-*
```

## ⏰ Part 3 — SLM (Snapshot Lifecycle Management)

### What SLM Does

```
SLM = Automated snapshot creation + retention management

Without SLM: Manual snapshots → someone forgets → data loss
With SLM:    Automatic daily/hourly snapshots → auto-delete old ones
```

### Production SLM Policy — Daily Snapshots

```bash
PUT _slm/policy/daily-snapshots
{
  "name": "<daily-snap-{now/d}>",
  "schedule": "0 30 1 * * ?",
  "repository": "s3-production-repo",
  "config": {
    "indices": ["*"],
    "include_global_state": true,
    "ignore_unavailable": true,
    "metadata": {
      "policy": "daily-snapshots",
      "environment": "production"
    }
  },
  "retention": {
    "expire_after": "30d",
    "min_count": 5,
    "max_count": 50
  }
}
```

### Production SLM Policy — Hourly for Critical Indices

```bash
PUT _slm/policy/hourly-critical-snapshots
{
  "name": "<hourly-critical-{now/H}>",
  "schedule": "0 0 * * * ?",
  "repository": "s3-production-repo",
  "config": {
    "indices": ["logs-payment-*", "logs-orders-*"],
    "include_global_state": false,
    "ignore_unavailable": true,
    "metadata": {
      "policy": "hourly-critical",
      "rpo": "1h"
    }
  },
  "retention": {
    "expire_after": "7d",
    "min_count": 24,
    "max_count": 168
  }
}
```

### Cron Schedule Reference

```
"0 30 1 * * ?"   → Every day at 01:30 UTC
"0 0 * * * ?"    → Every hour at :00
"0 0 */6 * * ?"  → Every 6 hours
"0 0 0 * * ?"    → Every day at midnight
"0 0 0 ? * MON"  → Every Monday at midnight

Format: seconds minutes hours day-of-month month day-of-week year
```

### SLM Management Commands

```bash
# List all SLM policies
GET _slm/policy

# Get specific policy
GET _slm/policy/daily-snapshots

# Execute policy NOW (manual trigger)
POST _slm/policy/daily-snapshots/_execute

# Get SLM stats
GET _slm/stats

# Get policy execution history
GET _slm/policy/daily-snapshots

# Response includes:
# last_success: { snapshot_name, time, details }
# last_failure: { snapshot_name, time, details }
# stats.snapshots_taken
# stats.snapshots_deleted
# stats.snapshots_failed

# Delete SLM policy (does NOT delete existing snapshots)
DELETE _slm/policy/daily-snapshots

# Start/Stop SLM globally
POST _slm/start
POST _slm/stop
GET _slm/status
```

### Retention Logic Explained

```
retention:
  expire_after: "30d"    → delete snapshots older than 30 days
  min_count: 5           → ALWAYS keep at least 5 snapshots (even if older than 30d)
  max_count: 50          → NEVER keep more than 50 snapshots (delete oldest first)

Real scenario:
  You have 55 snapshots, 10 older than 30 days
  → max_count=50: delete 5 oldest first
  → expire_after=30d: delete remaining 5 old ones
  → min_count=5: if only 5 left, stop deleting even if older than 30d
```

## 🔄 Part 4 — Restore

### Basic Restore

```bash
# Restore ALL indices from snapshot
POST _snapshot/s3-production-repo/snapshot-2026-04-27/_restore
{
  "include_global_state": false,
  "include_aliases": true
}

# Restore specific indices
POST _snapshot/s3-production-repo/snapshot-2026-04-27/_restore
{
  "indices": ["logs-nginx-production", "logs-api-production"],
  "include_global_state": false,
  "ignore_unavailable": true
}
```

### Restore with Rename (avoid conflict with existing indices)

```bash
POST _snapshot/s3-production-repo/snapshot-2026-04-27/_restore
{
  "indices": ["logs-nginx-production"],
  "rename_pattern": "logs-(.+)",
  "rename_replacement": "restored-logs-$1",
  "include_global_state": false
}
# Creates: restored-logs-nginx-production
# Original: logs-nginx-production (untouched)
```

### Restore Data Stream

```bash
# Close existing data stream first (if it exists)
POST logs-nginx-production/_close

# Restore
POST _snapshot/s3-production-repo/snapshot-2026-04-27/_restore
{
  "indices": ["logs-nginx-production"],
  "include_global_state": false
}

# Or restore to different name
POST _snapshot/s3-production-repo/snapshot-2026-04-27/_restore
{
  "indices": [".ds-logs-nginx-production-2026.04.01-000001"],
  "rename_pattern": ".ds-logs-nginx-production-(.+)",
  "rename_replacement": ".ds-restored-logs-nginx-$1"
}
```

### Monitor Restore Progress

```bash
# Check recovery progress
GET _cat/recovery?v&active_only=true

# Detailed recovery info
GET _recovery?human=true&active_only=true

# Check specific index restore
GET logs-nginx-production/_recovery
```

### Partial Restore (when some shards failed in snapshot)

```bash
POST _snapshot/s3-production-repo/snapshot-partial-2026-04-27/_restore
{
  "indices": ["logs-nginx-production"],
  "partial": true,
  "include_global_state": false
}
# partial: true → restore even if some shards are missing
# Use only when you accept partial data loss
```

## 🧊 Part 5 — Searchable Snapshots

### What Are Searchable Snapshots?

```
Traditional restore:
  Snapshot in S3 → Download ALL data to local disk → Search

Searchable Snapshot:
  Snapshot in S3 → Mount as index → Search (data fetched on demand from S3)
  → No local disk needed for data!
  → 90%+ cost reduction for cold/frozen data

Two modes:
  FULL COPY (cold tier):
    → Downloads data to local cache on first access
    → Fast repeated queries
    → Some local disk needed

  SHARED CACHE (frozen tier):
    → Shared cache across all frozen indices
    → Slower (fetches from S3 each time)
    → Minimal local disk
    → Cheapest option
```

### Mount a Searchable Snapshot (Cold Tier)

```bash
# Mount as full copy (cold tier)
POST _snapshot/s3-production-repo/snapshot-2026-04-01/_mount
{
  "index": ".ds-logs-nginx-production-2026.04.01-000001",
  "renamed_index": "restored-logs-nginx-2026-04-01",
  "index_settings": {
    "index.number_of_replicas": 0
  },
  "ignore_index_settings": ["index.refresh_interval"]
}
```

### Mount as Frozen (Cheapest — Shared Cache)

```bash
POST _snapshot/s3-production-repo/snapshot-2026-04-01/_mount?storage=shared_cache
{
  "index": ".ds-logs-nginx-production-2026.04.01-000001",
  "renamed_index": "frozen-logs-nginx-2026-04-01",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}
```

### Searchable Snapshot via ILM (Production Way)

```bash
# In ILM policy — frozen phase handles this automatically
PUT _ilm/policy/logs-with-searchable-snapshot
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": { "max_age": "1d", "max_primary_shard_size": "50gb" }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "searchable_snapshot": {
            "snapshot_repository": "s3-production-repo",
            "force_merge_index": true
          }
        }
      },
      "frozen": {
        "min_age": "90d",
        "actions": {
          "searchable_snapshot": {
            "snapshot_repository": "s3-production-repo"
          }
        }
      },
      "delete": {
        "min_age": "365d",
        "actions": { "delete": {} }
      }
    }
  }
}
```

### Searchable Snapshot — Required Node Config

```yaml
# elasticsearch.yml on frozen nodes
xpack.searchable.snapshot.shared_cache.size: 100gb
# OR as percentage of disk:
xpack.searchable.snapshot.shared_cache.size: 90%
```

### Cost Comparison — Real Numbers

```
90 TB dataset — 1 year retention:

All-hot (SSD):
  → $28,222/month

Hot (30d) + Warm (60d) + Cold searchable snapshot:
  → ~$8,000/month

Hot (30d) + Warm (30d) + Cold (30d) + Frozen searchable snapshot:
  → $3,290/month  (88% saving vs all-hot!)
```

## 🏭 Part 6 — Production Disaster Recovery Runbook

### RPO & RTO Targets

```
RPO (Recovery Point Objective) = How much data can you lose?
  → Hourly snapshots = max 1 hour data loss
  → Daily snapshots = max 24 hours data loss

RTO (Recovery Time Objective) = How fast must you recover?
  → Full restore: depends on data size + network
  → Searchable snapshot: near-instant (no download needed!)
  → 1TB restore from S3 ≈ 30-60 minutes (with 500mb/s throttle)
```

### Scenario 1: Single Index Corrupted

```bash
# Step 1: Close corrupted index
POST corrupted-index/_close

# Step 2: Find latest good snapshot
GET _snapshot/s3-production-repo/_all?sort=start_time&order=desc&size=5

# Step 3: Restore to temp name
POST _snapshot/s3-production-repo/daily-snap-2026-04-26/_restore
{
  "indices": ["corrupted-index"],
  "rename_pattern": "(.+)",
  "rename_replacement": "restored-$1",
  "include_global_state": false
}

# Step 4: Verify data
GET restored-corrupted-index/_count
GET restored-corrupted-index/_search?size=5

# Step 5: Switch alias
POST _aliases
{
  "actions": [
    { "remove": { "index": "corrupted-index", "alias": "my-alias" } },
    { "add": { "index": "restored-corrupted-index", "alias": "my-alias" } }
  ]
}
```

### Scenario 2: Full Cluster Recovery

```bash
# Step 1: Start fresh cluster with same version
# Step 2: Register same repository (read-only to avoid conflicts)
PUT _snapshot/s3-production-repo
{
  "type": "s3",
  "settings": {
    "bucket": "my-es-snapshots-prod",
    "readonly": true
  }
}

# Step 3: List available snapshots
GET _snapshot/s3-production-repo/_all?sort=start_time&order=desc

# Step 4: Restore latest snapshot with global state
POST _snapshot/s3-production-repo/daily-snap-2026-04-27/_restore
{
  "indices": ["*"],
  "include_global_state": true,
  "include_aliases": true
}

# Step 5: Monitor recovery
GET _cat/recovery?v&active_only=true
GET _cluster/health?wait_for_status=yellow&timeout=300s
```

### Scenario 3: Cross-Region DR

```bash
# Primary region: ap-south-1 (Mumbai)
# DR region: ap-southeast-1 (Singapore)

# On DR cluster: Register primary S3 bucket as READ-ONLY
PUT _snapshot/primary-s3-repo
{
  "type": "s3",
  "settings": {
    "bucket": "my-es-snapshots-prod",
    "region": "ap-south-1",
    "readonly": true,
    "base_path": "elasticsearch/production"
  }
}

# Restore from primary cluster's snapshots
POST _snapshot/primary-s3-repo/daily-snap-2026-04-27/_restore
{
  "indices": ["*"],
  "include_global_state": false
}
```

## ❌ Part 8 — Common Mistakes & Fixes

| Mistake | Impact | Fix |
|---|---|---|
| Storing credentials in API body | Security breach | Always use keystore: `elasticsearch-keystore add` |
| Single repository used by multiple clusters for WRITE | Snapshot corruption | One cluster writes, others read-only |
| Not verifying repository after creation | Silent failures | Always run `POST _snapshot/repo/_verify` |
| `include_global_state: true` on restore to different cluster | Cluster settings overwritten | Use `false` unless full cluster restore |
| Restoring over existing open index | Error — must close first | `POST index/_close` then restore |
| Taking snapshot during rolling restart | Partial/failed snapshot | Wait for cluster green, then snapshot |
| Not testing restore regularly | False confidence | Run quarterly restore drills |
| Deleting snapshots manually from S3 | Repository corruption | Always use ES DELETE API, never S3 console delete |
| No retention policy | S3 costs explode | Always configure SLM retention |
| Only daily snapshots for critical data | 24h data loss risk | Use hourly SLM for critical indices |
| `max_snapshot_bytes_per_sec` not set | Snapshot saturates network | Set to 40-60% of available bandwidth |

## 💡 Key Principles — Summary

```
1. Repository = storage, Snapshot = backup, SLM = automation

2. Snapshots are incremental after the first one
   → Daily snapshots don't cost 365x the first snapshot

3. NEVER write to same repository from two clusters
   → Repository corruption guaranteed

4. ALWAYS verify repository after creation
   → POST _snapshot/repo/_verify

5. Test your restores quarterly
   → A backup never tested is not a backup

6. Use SLM for automation — never rely on manual snapshots alone

7. Searchable snapshots = 90% cost reduction for cold/frozen data
   → No need to restore to local disk for old data

8. Set throttle limits to protect production workloads
   → max_snapshot_bytes_per_sec / max_restore_bytes_per_sec

9. Use rename_pattern on restore to avoid conflicts
   → Never restore over existing open indices

10. include_global_state: true only for full cluster recovery
    → Overwrites cluster settings, templates, ILM policies
```
