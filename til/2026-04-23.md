## 🖥️ The 4 Computing Resources ES Cares About

Before any sizing math, understand what ES actually consumes:

| Resource | How ES Uses It | Key Rule |
|---|---|---|
| **Storage** | Persists segments, translog, replicas | SSD for hot, HDD for warm, snapshots for cold/frozen |
| **Memory** | JVM Heap + OS Cache | 50% RAM for heap (max 30GB), rest for OS cache |
| **Compute** | Thread pools for indexing/search | More CPU cores = more parallelism |
| **Network** | Replication, CCS, ingest | Rarely a bottleneck unless massive CCS deployments |

```
💡 OS Cache is often underrated.
   ES avoids disk reads entirely for search/aggs if data fits in OS cache.
   This is why memory:data ratio matters more than raw storage.
```

## 🏗️ Node Roles — Size Each Role Separately

| Role | CPU | Memory | Storage | Key Point |
|---|---|---|---|---|
| **Data (Hot)** | High | High | SSD DAS | Workhorse — size this first |
| **Data (Warm)** | Medium | Medium | HDD DAS | Cheaper, less active indexing |
| **Data (Cold/Frozen)** | Low | Low | Searchable Snapshots | No replicas needed |
| **Master** | Low | Low | Minimal | Stability — never mix with data |
| **Ingest** | High | Medium | Minimal | Heavy pipeline processing |
| **Coordinator** | Low | Medium | None | Offloads search merge from data nodes |
| **ML** | Extreme | Extreme | Low | Isolate always |

## 🔢 The 3 Limits That Actually Matter (2026 Updated!)

### Limit 1 — Shard Size: 10 GB to 50 GB per shard

```
Too small → Master node overhead, heap pressure, network chatter
Too large → Slow queries, slow recovery after node failure

✅ Target: 10–50 GB per primary shard
✅ Max documents per shard: 200 million
✅ ILM rollover trigger: 50 GB (official recommended default)

❌ OLD RULE (retired in ES 8.3): "20 shards per GB of heap"
✅ NEW RULE: Watch shard SIZE (10-50GB) + shard COUNT per node (below)
```

```bash
# Check shard sizes
GET _cat/shards?v&h=index,shard,prirep,state,store&s=store:desc

# Check index sizes
GET _cat/indices?v&h=index,pri,rep,docs.count,store.size,pri.store.size&s=store.size:desc
```

### Limit 2 — Shard Budget: 1,000 shards per non-frozen data node

```
ILM silently creates shards every rollover. Do the math!

Example:
  Daily rollover × 5 primary shards × 1 replica = 10 shards/day
  → 1 node fills its budget in ~100 days without any changes!

Fix options:
  → Wider rollover interval (weekly/monthly if <50GB/day)
  → Fewer primaries per index (1-2 for smaller volumes)
  → Add more data nodes
```

```bash
# Check shards per node
GET _cat/nodes?v&h=name,shards,disk.used,heap.percent

# Master node heap planning
# Rule: 1 GB heap per 3,000 indices on master nodes
```

### Limit 3 — Storage Tier Mismatch

```
Keeping data on hot (expensive SSD) longer than needed = 💸 waste

Real benchmark (90 TB dataset):
  All-hot  →  $28,222/month
  Hot+Frozen → $3,290/month   (90% cost reduction!)
```

| Tier | Storage Type | Memory:Data Ratio | Use When |
|---|---|---|---|
| Hot | Local SSD (DAS) | 1:30 | Active indexing + frequent search |
| Warm | HDD (DAS) | 1:100 to 1:160 | Infrequent search, no active indexing |
| Cold | Searchable Snapshots | 1:500+ | Rare access, ~50% cheaper than warm |
| Frozen | Searchable Snapshots | 1:500+ | Archives, up to 20x cheaper than warm |

## 📐 The Sizing Math — Volume Method

**Discovery Questions to Answer First:**
- How much raw data (GB) per day?
- How many days of retention? (Hot vs Warm separately)
- How many replicas?
- How much RAM per data node?

**The Formulas:**

```
Step 1 — Total Data
Total Data (GB) = Raw GB/day × Days retained × Expansion Factor × (Replicas + 1)

Expansion Factor (average) ≈ 1.2x
  (JSON verbosity + indexing structures − compression)
  Measure your own: index_size_without_replica ÷ raw_volume_size

Step 2 — Total Storage (add headroom)
Total Storage (GB) = Total Data × (1 + 0.15 watermark + 0.10 margin)
                                        ↑ disk watermark    ↑ safety buffer

Step 3 — Total Data Nodes
Total Data Nodes = ROUNDUP(Total Storage / RAM per node / Memory:Data ratio) + 1
                                                                                 ↑ always add 1 for failover
```

## 📊 Worked Example — Small Cluster

```
Input:
  Raw data: 1 GB/day
  Retention: 9 months (270 days)
  Replicas: 1
  RAM per node: 8 GB
  Memory:Data ratio (hot): 1:30

Calculation:
  Total Data    = 1 × 270 × 1.2 × 2 = 648 GB
  Total Storage = 648 × 1.25        = 810 GB
  Data Nodes    = ROUNDUP(810 / 8 / 30) + 1 = 3 + 1 = 4 nodes
```

## 📊 Worked Example — Large Cluster (Hot + Warm)

```
Input:
  Raw data: 100 GB/day
  Hot retention: 30 days
  Warm retention: 12 months (365 days)
  Replicas: 1
  RAM per node: 64 GB

HOT TIER:
  Total Data    = 100 × 30 × 1.2 × 2  = 7,200 GB
  Total Storage = 7,200 × 1.25        = 9,000 GB
  Data Nodes    = ROUNDUP(9000/64/30) + 1 = 5 + 1 = 6 hot nodes

WARM TIER:
  Total Data    = 100 × 365 × 1.2 × 2 = 87,600 GB
  Total Storage = 87,600 × 1.25       = 109,500 GB
  Data Nodes    = ROUNDUP(109500/64/160) + 1 = 11 + 1 = 12 warm nodes
```

## ⚠️ Common Sizing Mistakes

| Mistake | Impact | Fix |
|---|---|---|
| Not measuring actual expansion factor | Over/under provisioning | Index sample data, measure ratio |
| Ignoring shard count as ILM runs | Node hits 1000-shard ceiling | Calculate shards/day × retention days |
| NAS for hot data | Latency, possible corruption | Always use local DAS SSD for hot |
| Mixing master + data roles | Master instability under load | Always use dedicated masters (3 nodes) |
| Skipping replica in math | Storage underestimate | Always multiply by (replicas + 1) |
| Using old "20 shards/GB heap" rule | Unnecessary node additions | Retired in 8.3 — use 10-50GB shard size |

## 💡 Key Sizing Principles — Summary

```
1. Measure before you plan  → Get real expansion factor from sample data
2. Size tiers independently → Hot, Warm, Cold each have different ratios
3. Always +1 node           → Failover capacity is not optional
4. Always +25% storage      → 15% watermark + 10% margin
5. Shard size 10-50 GB      → The one rule that replaced everything else
6. 1000 shards/node limit   → Calculate ILM shard creation rate!
7. Local SSD for hot        → Never NAS, never network storage
8. Benchmark with Rally     → Math gives you a starting point, not a guarantee
```
